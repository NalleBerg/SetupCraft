================================================================================
  Focus Restore on App Activation  —  INTERNALS Reference
================================================================================

PURPOSE
-------
When the user Alt+Tabs back to the app (or clicks it on the taskbar), the active
text selection should immediately be live (shown in the system highlight colour)
so the user can press Ctrl+C / Ctrl+X without first clicking inside the editor.

Without a fix, Windows gives keyboard focus to the top-level frame window, not
to any child editor, so the selection appears greyed out and unusable until the
user clicks.

================================================================================
THE PROBLEM WITH WM_ACTIVATE
================================================================================

A naive first attempt handles WM_ACTIVATE:

    case WM_ACTIVATE:
        if (LOWORD(wParam) != WA_INACTIVE)
            SetFocus(hChildEditor);          // ← does NOT stick
        return DefWindowProcW(hwnd, msg, wParam, lParam);

This does not work.  DefWindowProcW's handling of WM_ACTIVATE internally calls
SetFocus on the top-level window itself, which runs AFTER our SetFocus call and
silently overrides it.  The child editor never actually receives focus.

Reversing the order (call DefWindowProcW first, then SetFocus) is also fragile
because Windows can still redirect focus during the activation sequence.

================================================================================
THE CORRECT PATTERN: WM_SETFOCUS on the top-level window
================================================================================

Handle WM_SETFOCUS on the main frame window and redirect to the active editor:

    case WM_SETFOCUS: {
        // Redirect focus from the frame to whichever child editor is active.
        NeTabDoc* doc = NeTabs_GetActiveDoc(hwnd);
        if (doc) {
            HWND hTarget = doc->hSci ? doc->hSci : doc->hEdit;
            if (hTarget) SetFocus(hTarget);
        }
        return 0;
    }

Why this works
--------------
The activation sequence for a top-level window is:

  1. WM_NCACTIVATE     — non-client area redraws (title bar colour change)
  2. WM_ACTIVATE       — DefWindowProcW calls SetFocus(hwnd) here
  3. WM_SETFOCUS       — delivered to hwnd because step 2 gave it focus

By the time WM_SETFOCUS arrives, all of Windows' internal activation machinery
is done.  Calling SetFocus(childEditor) from here is the last word; nothing
overrides it.  The child editor receives WM_SETFOCUS and the selection turns
blue immediately.

IMPORTANT — multi-editor-type tabs
------------------------------------
If the app has tabs that use different child editor controls (e.g. RichEdit for
RTF documents and Scintilla for plain text / code), you MUST redirect to the
correct one.  A helper like NeTabs_GetActiveEdit() that always returns the
RichEdit handle will silently set focus to an invisible window when a Scintilla
tab is active, leaving the selection grey.

Always inspect the active doc and branch on which control is visible:

    HWND hTarget = doc->hSci ? doc->hSci   // Scintilla tab is active
                             : doc->hEdit;  // RichEdit  tab is active

Side effects to be aware of
----------------------------
* Every time the frame window receives WM_SETFOCUS (not just on Alt+Tab), this
  code runs.  That is intentional and harmless: focus clicking on an empty part
  of the toolbar or a disabled area will correctly redirect to the editor.
* The handler returns 0 without calling DefWindowProcW.  That is correct;
  DefWindowProcW does nothing useful for WM_SETFOCUS on an overlapped window.
* Do NOT call this from WM_ACTIVATE.  WM_SETFOCUS fires at exactly the right
  moment; WM_ACTIVATE fires too early.

================================================================================
PREVENTING DIALOGS FROM STEALING FOCUS BACK FROM OTHER APPS
================================================================================

Problem
-------
Some apps show a dialog in response to an event (e.g. "file changed on disk",
"file saved", "connection timeout") that fires when the editor regains focus.
If the dialog's close code calls SetForegroundWindow(parent) unconditionally,
it yanks focus back to the app even when the user has already Alt+Tabbed to
something else — a browser, another editor, whatever.

This is doubly bad when the dialog has an auto-close countdown timer, because
the sequence is:
  1. User saves file via FTP, then immediately Alt+Tabs to browser.
  2. FTP "file saved" dialog is still open in the background.
  3. Auto-close fires, calls SetForegroundWindow(NSBEdit) → browser loses focus.

Guard pattern: only call SetForegroundWindow if you still own the foreground
-----------------------------------------------------------------------------
In the dialog close / destroy path, check the current foreground window before
restoring focus to the parent:

    // In WM_DESTROY or just before EndDialog / DestroyWindow:
    if (parent && IsWindow(parent)) {
        EnableWindow(parent, TRUE);        // re-enable the parent first
        HWND fg = GetForegroundWindow();
        if (fg == dlg || fg == parent || IsChild(parent, fg))
            SetForegroundWindow(parent);   // we still own the foreground — safe
        // else: user has moved to another app — leave them there
    }

Without this guard, every modal dialog in the app is a potential focus-thief.

Auto-close timeout consideration
----------------------------------
If the dialog auto-closes after a countdown (e.g. "File saved — 3 … 2 … 1"),
set the timeout short enough (≤ 1 000 ms) that the window disappears before a
typical Alt+Tab sequence completes.  A 2 500 ms countdown is long enough for
the user to switch apps; then the countdown fires and steals focus back.

================================================================================
PREVENTING BACKGROUND DIALOGS ON APP RE-ACTIVATION (WM_ACTIVATEAPP)
================================================================================

Problem
-------
Some in-app dialogs are triggered by focus events on a child editor — the most
common example being "file changed on disk" detection fired by EN_SETFOCUS or
SCN_FOCUSIN.  If the user Alt+Tabs away and quickly back, the focus event fires
but the dialog should not appear (the app is still "re-activating" and the
dialog would arrive as an unexpected pop-up).

Basic fix: WM_ACTIVATEAPP flag
-------------------------------
Track whether the app is actually in the foreground:

    // near the top of the file (file scope):
    static bool s_appIsActive = true;   // true = our app owns the foreground

    // in WM_ACTIVATEAPP:
    case WM_ACTIVATEAPP:
        s_appIsActive = (wParam != 0);
        return DefWindowProcW(hwnd, msg, wParam, lParam);

In the focus-triggered check:

    void Ne_CheckExternalFileChangeOnFocus(HWND hwnd) {
        if (!s_appIsActive) return;     // not our foreground — do nothing
        // … rest of the check …
    }

The Chromium double-bounce problem
------------------------------------
This naive fix is insufficient for Chromium-based browsers (Chrome, Opera,
Edge, Brave).  When the user switches TO a Chromium window (e.g. pressing
Ctrl+R for a page refresh), Chromium briefly fires WM_ACTIVATEAPP(TRUE) back
to the PREVIOUS foreground window as part of its own internal activation
settling.  The sequence:

    1. User presses Alt+Tab → browser.
       → Our app receives WM_ACTIVATEAPP(FALSE)  (s_appIsActive = false)
    2. Browser activates; ~50–200 ms later, Chromium fires WM_ACTIVATEAPP(TRUE)
       to our window  (spurious bounce).
       → s_appIsActive = true  (WRONG — we don't own the foreground)
    3. User presses Ctrl+R in the browser.
       → EN_SETFOCUS fires → s_appIsActive is true → dialog pops up in the bg.
    4. Dialog closes → SetForegroundWindow(our app) → browser loses focus.

Fix: debounce on WM_ACTIVATEAPP(FALSE)
----------------------------------------
Record the tick count every time the app is deactivated.  Suppress the
focus-event check for a short window after the last deactivation:

    static bool  s_appIsActive   = true;
    static DWORD s_deactivatedAt = 0;      // GetTickCount() when last deactivated

    case WM_ACTIVATEAPP:
        s_appIsActive = (wParam != 0);
        if (!s_appIsActive)
            s_deactivatedAt = GetTickCount();
        return DefWindowProcW(hwnd, msg, wParam, lParam);

    void CheckOnFocus(HWND hwnd) {
        if (!s_appIsActive) return;
        if (GetTickCount() - s_deactivatedAt < 500) return;  // debounce 500 ms
        // … rest of the check …
    }

Why 500 ms?  Chromium's spurious WM_ACTIVATEAPP(TRUE) arrives within ~50–200 ms
of the deactivation.  500 ms is long enough to outlast any Chromium bounce yet
short enough that a genuine re-activation (user Alt+Tabs back and waits) will
pass the check within half a second.

NOTE: the debounce fires even when s_appIsActive is true (after Chromium's
spurious bounce re-sets it).  The GetTickCount comparison is the real guard;
s_appIsActive is a secondary quick-reject only.

================================================================================
WHY THE "CHECK FG BEFORE SetForegroundWindow" GUARD BREAKS FOR MODAL DIALOGS
================================================================================

The guard pattern shown in the previous section has a subtle flaw when applied
to a custom modal dialog built with a manual message loop:

    while (IsWindow(dlg) && GetMessageW(&msg, NULL, 0, 0) > 0) { ... }

The loop condition is IsWindow(dlg).  When the user clicks a button, the handler
calls DestroyWindow(dlg).  The WndProc returns, the next GetMessageW reads any
remaining queued messages, and eventually IsWindow(dlg) becomes FALSE — at which
point the loop exits.

CRITICAL: by the time the loop exits, dlg is already destroyed.  Any code that
runs after the loop — including the "check fg" teardown — is operating on an
invalid handle.

    HWND fg = GetForegroundWindow();
    EnableWindow(parent, TRUE);
    if (fg == dlg || ...)          // ← dlg is INVALID here — ALWAYS FALSE
        SetForegroundWindow(parent);

Two failures occur:
  1. fg == dlg is always false (handle recycled or NULL — GetForegroundWindow
     never returns a destroyed HWND).
  2. EnableWindow(parent, TRUE) itself can silently hand focus to the next
     window in Z-order (e.g. Explorer, the taskbar) before we even call
     SetForegroundWindow.  So fg == parent may also be false by then.

The result: SetForegroundWindow is never called, and the parent window sits
unfocused behind whatever grabbed focus from EnableWindow.

================================================================================
THE TOPMOST TRICK — RELIABLE PARENT RESTORE AFTER MODAL DIALOG
================================================================================

Overview
--------
The "TOPMOST trick" is the reliable Win32 pattern for restoring focus to a
parent window after its modal child is destroyed.  It works by briefly making
the parent a topmost window — which forces it to the top of the Z-order and
gives it foreground priority — then immediately removing the topmost flag so it
reverts to a normal, non-always-on-top window.

Code (insert in the modal teardown, after the message loop):

    if (parent && IsWindow(parent)) {
        EnableWindow(parent, TRUE);          // re-enable first

        if (dd.restoreOnClose) {
            // TOPMOST trick: set topmost to force focus, immediately remove it.
            // The window stays in the foreground as a normal (non-topmost) window.
            SetWindowPos(parent, HWND_TOPMOST,
                         0, 0, 0, 0, SWP_NOMOVE | SWP_NOSIZE);
            SetWindowPos(parent, HWND_NOTOPMOST,
                         0, 0, 0, 0, SWP_NOMOVE | SWP_NOSIZE);
        }
    }

Why two SetWindowPos calls?
---------------------------
SetWindowPos(HWND_TOPMOST) inserts the window at the very top of the Z-order
AND calls SetForegroundWindow internally — it is one of the few Win32 APIs that
can reliably steal foreground status from another process.  This handles the
case where EnableWindow(parent, TRUE) handed focus to an unwanted window.

The immediate SetWindowPos(HWND_NOTOPMOST) removes the WS_EX_TOPMOST style
without changing the Z-order relative to non-topmost windows.  The parent is
still in the foreground — it just is no longer pinned above every other window.

restoreOnClose — the creation-time guard
-----------------------------------------
Because the dialog is open for only ~1 second (auto-close or Enter to dismiss),
the risk of the user switching apps WHILE the dialog is open is low.  However,
if the dialog were triggered while the app is in the background, blindly
applying the TOPMOST trick would steal focus from whatever app is in the front.

The guard: record at dialog CREATION time whether the parent was the foreground
window.  Only apply the TOPMOST trick if it was:

    // --- before CreateWindowExW ---
    dd.restoreOnClose = (parent && IsWindow(parent) &&
                         GetForegroundWindow() == parent);

This is captured before EnableWindow(parent, FALSE) and before the dialog
appears.  If the user is in another app when the dialog somehow fires, the flag
is false and no focus grab occurs.

When NOT to use the guard (dialogues always triggered by user action)
----------------------------------------------------------------------
If the dialog is always triggered synchronously by a user gesture (button click,
menu, keyboard shortcut), the parent IS guaranteed to be the foreground window at
that moment — the user just interacted with it.  In that case you can set
restoreOnClose unconditionally (= true) or skip the flag entirely.

================================================================================
SUMMARY
================================================================================

  Wrong: SetFocus in WM_ACTIVATE   → overridden by DefWindowProcW
  Wrong: SetFocus to stale/hidden child window (e.g. RichEdit when Scintilla
         tab is active)            → focus goes nowhere visible
  Right: WM_SETFOCUS on the top-level frame → redirect to the currently visible
         child editor (hSci or hEdit depending on the active tab)

  Wrong: SetForegroundWindow(parent) unconditionally in dialog close code
         → steals focus back from other apps after auto-close countdowns
  Wrong: "check fg == dlg" guard after modal loop exits
         → dlg handle is already destroyed when the loop exits; guard is
            always false; EnableWindow(parent,TRUE) may also steal focus
  Right: use the TOPMOST trick at close time — SetWindowPos(HWND_TOPMOST)
         then SetWindowPos(HWND_NOTOPMOST) — reliably restores focus to the
         parent without permanently pinning it.  Gate on restoreOnClose flag
         set at creation time (was parent the foreground when dialog opened?).

  Wrong: checking s_appIsActive alone (naive WM_ACTIVATEAPP flag)
         → Chromium fires a spurious WM_ACTIVATEAPP(TRUE) bounce (~50–200 ms
            after deactivation) that resets the flag incorrectly
  Right: combine the flag with a 500 ms debounce on s_deactivatedAt so the
         focus-triggered check is suppressed for 500 ms after any deactivation
