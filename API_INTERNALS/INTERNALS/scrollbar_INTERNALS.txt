  Shortcuts-page vertical scrollbar — architecture, failure modes, and rationale
================================================================================

1. OVERVIEW
───────────
The Shortcuts page (page index 2) uses a custom hidden scrollbar (my_scrollbar,
HMSB s_hMsbSc) attached to the main window — identical to the Dialogs page
pattern (s_hMsbIdlg).  There is NO native WS_VSCROLL on the main window.
SCROLLINFO is used only as an internal range tracker; the visible widget is the
MSB bar.  Scrolling is driven by WM_VSCROLL (MSB thumb drag / arrow) and
WM_MOUSEWHEEL.

A canonical abstraction is provided in shortcuts.cpp: the current scroll
offset in pixels is stored in s_scScrollOffset and exposed via:

    void SC_SetScrollOffset(int off)
    int  SC_GetScrollOffset()

SC_RefreshDesktopStrip and SC_RefreshPinStrips subtract the current offset when
positioning their controls, so those helpers are scroll-aware without needing
any extra wiring.

[Migration note — 2026-04-05]
Previously a native WS_VSCROLL was conditionally added to the main window when
contentH > viewH.  Replaced with always-attached MSB (same as Dialogs page) so
the bar fades in/out gracefully and no FRAMECHANGED flash occurs on page build.


2. SCROLLINFO SETUP  (mainwindow.cpp, SwitchPage case 2)
─────────────────────────────────────────────────────────
Called once after SC_BuildPage returns the absolute-Y of the last content row:

    s_scPageContentH = SC_BuildPage(...);   // absolute Y in main-window coords

    int pageY   = s_toolbarHeight;
    int statusH = <measured via GetWindowRect(s_hStatus)>;
    int viewH   = clientBottom - pageY - statusH;
    int contentH = s_scPageContentH - pageY + S(15);   // + 15 px bottom pad

    SCROLLINFO si = {};
    si.cbSize = sizeof(si);
    si.fMask  = SIF_RANGE | SIF_PAGE | SIF_POS | SIF_DISABLENOSCROLL;
    si.nMin   = 0;
    si.nMax   = std::max(contentH - 1, viewH);
    si.nPage  = viewH;
    si.nPos   = 0;
    SetScrollInfo(hwnd, SB_VERT, &si, FALSE);   // FALSE = no native bar

    if (!s_hMsbSc) s_hMsbSc = msb_attach(hwnd, MSB_VERTICAL);
    else            msb_sync(s_hMsbSc);
    msb_set_insets(s_hMsbSc, pageY, statusH);   // clamp bar to page area
    msb_set_edge_gap(s_hMsbSc, 4);              // 4 px inward from right edge

    maxSc (effective) = contentH - viewH = s_scPageContentH + S(15) - pageY
                        - (clientBottom - pageY - statusH)
                      = s_scPageContentH + S(15) - clientBottom + statusH

At pos == maxSc the logical bottom of content (s_scPageContentH + S(15))
aligns exactly with the top of the status bar.


3. WHY SW_SCROLLCHILDREN DOES NOT WORK
────────────────────────────────────────
ScrollWindowEx(hwnd, 0, -dy, prcScroll, ..., SW_SCROLLCHILDREN) moves only
child windows that **intersect** prcScroll at call time.

Problem: controls created by SC_BuildPage start at their absolute-Y positions.
When the page is taller than the viewport, some controls start BELOW the clip
rectangle {0, pageY, width, clientBottom-statusH}.  SW_SCROLLCHILDREN never
touches them — they stay at their initial Y and are permanently unreachable.

The status bar also sits exactly at the clip-rect boundary; depending on the
window's DPI and rounding it could be caught and moved down one pixel each
scroll step, corrupting its position.


4. CORRECT SCROLL MECHANISM
─────────────────────────────
Both WM_MOUSEWHEEL and WM_VSCROLL enumerate every direct child of hwnd and
call SetWindowPos on each page control individually.  The MSB bar window itself
is excluded from the loop (it must not be scrolled):

    HWND hMsbBar = s_hMsbSc ? msb_get_bar_hwnd(s_hMsbSc) : NULL;
    HWND hC = GetWindow(hwnd, GW_CHILD);
    while (hC) {
        HWND hN = GetWindow(hC, GW_HWNDNEXT);
        if (hC != s_hStatus && hC != s_hAboutButton && hC != hMsbBar) {
            int cid = GetDlgCtrlID(hC);
            bool isTB = (cid >= IDC_TB_FILES && cid <= IDC_TB_ABOUT) ||
                         cid == IDC_TB_DIALOGS || cid == IDC_TB_COMPONENTS ||
                         cid == IDC_TB_EXIT    || cid == IDC_TB_CLOSE_PROJECT;
            if (!isTB) {
                RECT rcC; GetWindowRect(hC, &rcC);
                MapWindowPoints(HWND_DESKTOP, hwnd, (POINT*)&rcC, 2);
                SetWindowPos(hC, NULL, rcC.left, rcC.top - dy, 0, 0,
                    SWP_NOZORDER | SWP_NOSIZE | SWP_NOACTIVATE);
            }
        }
        hC = hN;
    }

Enumeration includes controls that are CURRENTLY above, within, or below the
visible page area, so every control is repositioned correctly regardless of
where it started.

After moving children the status bar is restored to HWND_TOP so it paints last
and visually covers any page control whose window rect has scrolled into the
status bar zone:

    SetWindowPos(s_hStatus, HWND_TOP,
        0, clientBottom - statusH, clientWidth, statusH, SWP_NOACTIVATE);

Then the page area is invalidated:

    RECT pageRect = { 0, spY, clientWidth, clientBottom - statusH };
    InvalidateRect(hwnd, &pageRect, TRUE);
    UpdateWindow(s_hStatus);

SC_SetScrollOffset(newPos) is called last so SC_RefreshDesktopStrip /
SC_RefreshPinStrips produce correctly offset Y values on the next refresh.


5. WINDOWS TO EXCLUDE FROM SCROLLING
──────────────────────────────────────
Handle          Condition                   Reason
─────────────── ─────────────────────────── ──────────────────────────────────
s_hStatus       explicitly excluded         always pinned to window bottom
s_hAboutButton  explicitly excluded         top-right corner, always visible
hMsbBar         msb_get_bar_hwnd(s_hMsbSc)  the scrollbar widget itself
toolbar buttons IDC_TB_FILES … IDC_TB_ABOUT Fixed toolbar area; moving them
                IDC_TB_DIALOGS              would corrupt the toolbar layout.
                IDC_TB_COMPONENTS
                IDC_TB_EXIT
                IDC_TB_CLOSE_PROJECT

All other direct children of hwnd while page 2 is active are page controls
created by SC_BuildPage and are safe to move.


6. TEARDOWN  (SwitchPage in mainwindow.cpp + SC_TearDown in shortcuts.cpp)
────────────────────────────────────────────────────────────────────────────
SwitchPage teardown (before DestroyWindow loop):

    if (s_hMsbSc) { msb_detach(s_hMsbSc); s_hMsbSc = NULL; }

SC_TearDown (shortcuts.cpp) destroys strip controls and resets offset:

    DestroyWindow(DSK_STRIP)      — desktop strip control
    DestroyWindow(SMPIN_STRIP)    — Start-menu pin strip
    DestroyWindow(TBPIN_STRIP)    — Taskbar pin strip
    s_scScrollOffset = 0          — reset logical offset

No WS_VSCROLL removal is needed (no native bar was added).


7. SC_BUILDPAGE RETURN VALUE
──────────────────────────────
SC_BuildPage returns int — the absolute Y coordinate (main-window client space)
of the first pixel below the last control row, including any trailing padding
accumulated inside SC_BuildPage.

Callers store this in mainwindow.cpp: static int s_scPageContentH = 0;

The extra S(15) padding added at SCROLLINFO setup time provides visual breathing
room at the bottom of the content.


8. STATUS-BAR HEIGHT
──────────────────────
Always measured dynamically:

    RECT rcSB; GetWindowRect(s_hStatus, &rcSB);
    int statusH = rcSB.bottom - rcSB.top;

Using a hardcoded constant (e.g. 25 px) is wrong: the status bar height depends
on the current font and DPI and may not match the live window height.


9. KNOWN LIMITATIONS
──────────────────────
- There is no container child window.  All Shortcuts-page controls are direct
  children of the main window (hwnd).  This means their window rectangles can
  technically extend into the status-bar zone when scrolled; the HWND_TOP /
  UpdateWindow pattern prevents visual corruption but does not prevent hit-test
  noise in that overlap strip.
- Window resize while page 2 is active does not recalculate maxSc or reposition
  page controls; a page-switch round-trip is required to resize the page
  correctly.


10. START-MENU TREEVIEW MSBs  (s_hMsbScSmTreeV / s_hMsbScSmTreeH)
────────────────────────────────────────────────────────────────────
The Shortcuts page also has a Start-Menu TreeView (IDC_SC_SM_TREE,
s_hScStartMenuTree) which gets its own pair of custom scroll bars so it
matches the style of the Registry and Files tree views.

Creation style — REQUIRED:
    WS_CHILD | WS_VISIBLE | WS_VSCROLL | WS_HSCROLL | TVS_…

WS_VSCROLL and WS_HSCROLL MUST be set even though the native bar visuals are
hidden by MSB (ShowScrollBar FALSE).  Without them, the TreeView never
populates SCROLLINFO (nMax always stays 0) and the MSB bars are permanently
blank with no scrolling.

Attach/detach wiring follows the exact same pattern as the Registry page MSBs;
see scrollbar_reopen_INTERNALS.txt "RULE FOR ADDING MSBs" (points a–i) and
the "TREEVIEW/LISTVIEW SCROLLINFO RESET BUG" section for the ShowScrollBar
nMax-reset pitfall that is already handled inside my_scrollbar.cpp.

Initial visibility (SwitchPage case 2, after msb_attach + msb_set_edge_gap):
    RECT rcT; GetClientRect(hSmTree, &rcT);
    SendMessageW(hSmTree, WM_SIZE, SIZE_RESTORED, MAKELPARAM(rcT.right, rcT.bottom));
This sends WM_SIZE through the installed subclass.  origProc(WM_SIZE) causes the
TreeView to recompute its layout (nPage = items-per-view).  The subclass then
captures that fresh SCROLLINFO, hides native bars, restores, and calls the
UNGUARDED Msb_UpdateVisibility on both axes — correct INVISIBLE/HIDDEN from page open.

TVN_ITEMEXPANDED hook (mainwindow.cpp WM_NOTIFY, after SC_OnNotify dispatch):
    if (nmhdr->idFrom == IDC_SC_SM_TREE && nmhdr->code == TVN_ITEMEXPANDED
        && (s_hMsbScSmTreeV || s_hMsbScSmTreeH)) {
        // hide native bars, restore SCROLLINFO, then:
        msb_reposition(s_hMsbScSmTreeV);
        msb_reposition(s_hMsbScSmTreeH);
    }
Required because tree expansion/collapse doesn't trigger WM_VSCROLL or WM_SIZE
on the TreeView.  Guard "(V||H)" prevents running during initial populate (before
MSBs are attached), avoiding stale-SCROLLINFO corruption.

Edge gap:
    msb_set_edge_gap(s_hMsbScSmTreeH, SM_CYEDGE + SM_CYBORDER + 6);
Applied to the H-bar so it clears the WS_EX_CLIENTEDGE border at the bottom
of the tree.  The V-bar uses the default gap (0 — hugs the right edge).


11. DEPENDENCIES PAGE LISTVIEW MSBs  (s_hMsbDepListV / s_hMsbDepListH)
────────────────────────────────────────────────────────────────────────
IDC_DEP_LIST (deps.cpp) gets V+H Custom scrollbars so the Dependencies page
matches all other pages.

Key ListView H-axis notes (see scrollbar_custom_make_INTERNALS.txt §KEY NOTES):
  • GetScrollInfo(SB_HORZ).nPos is unreliable for ListView H: ShowScrollBar(FALSE)
    zeroes it; LVM_SCROLL doesn't update it.  MSB tracks position in MsbCtx.lvHPos.
  • H scroll range (nMax/nPage) uses live column widths + client width, NOT
    GetScrollInfo, because the native bar may have zeroed nMax.  Helper:
    Msb_FixListViewHScrollInfo (called in Msb_Layout, Msb_Scroll, Msb_ScrollToPos,
    track-click, drag).
  • WM_HSCROLL(SB_THUMBPOSITION) does NOT work for absolute positioning (requires
    SCROLLINFO.nTrackPos which only the native scrollbar machinery sets); use
    LVM_SCROLL(delta) where delta = newPos - lvHPos.
  • Arrows and tilt wheel use WM_HSCROLL (which the ListView handles correctly
    when nMax is pre-restored before the send).
  • H scrollbar must be suppressed explicitly after every InsertItem/populate call
    in deps.cpp using the capture/restore pattern (not plain ShowScrollBar):
        SCROLLINFO siH = {sizeof(siH), SIF_ALL};
        GetScrollInfo(s_hDepList, SB_HORZ, &siH);
        ShowScrollBar(s_hDepList, SB_HORZ, FALSE);
        if (siH.nMax > siH.nMin) SetScrollInfo(s_hDepList, SB_HORZ, &siH, FALSE);
        msb_sync(s_hMsbDepListH);

Teardown — MUST come BEFORE the controlIds[] DestroyWindow loop (same rule as
all other MSB-attached controls — see scrollbar_reopen_INTERNALS.txt):
    DEP_TearDown(hwnd);   // contains msb_detach calls
    // ← controlIds[] loop follows
