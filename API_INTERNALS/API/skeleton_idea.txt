================================================================================
  Project Skeleton — Idea & Checklist
================================================================================

  Goal: when SetupCraft is published, extract a reusable Win32/C++ project
  skeleton so future projects start with a solid, proven foundation rather
  than from scratch.

  Philosophy: API_list.txt is a living document — updated throughout
  development every time a module is added, moved, or improved. By the time
  this project is published, API_list.txt will be a complete, accurate index
  of every reusable tool and architectural pattern, ready to seed the skeleton.

  ── Notes & decisions to make ────────────────────────────────────────────────

  - Decide on a skeleton repo name (e.g. "win32-cpp-skeleton" or "WinBase").
  - Strip all SetupCraft-specific IDs, DB schema, and page logic — skeleton
    should compile to a blank main window + toolbar stub + status bar only.
  - Keep mingw_dlls\README.md so the DLL bundling convention is documented.
  - Consider a fix_locales.ps1 stub for future locale sync automation.
  - my_scrollbar is the heaviest dependency; decide whether it ships as
    source pair (my_scrollbar.h/.cpp) or as a header-only option.
  - ListView H-scroll lesson: GetScrollInfo.nPos is zeroed by
    ShowScrollBar(FALSE) and not updated by LVM_SCROLL.  The only
    reliable true offset is -Header_GetItemRect(col0).left (encapsulated
    as Msb_GetListViewHPos()).  WM_NCPAINT fires synchronously inside
    LVM_SCROLL before the header repositions — do NOT read or write lvHPos
    there.  The library intercepts LVM_DELETEALLITEMS, LVM_SETCOLUMNWIDTH,
    and HDN_ENDTRACK internally so callers need no post-call msb_reposition.
  - LVM_SCROLL delta-clamping pitfall (SOLVED v2026.05.02.10/11): LVM_SCROLL
    uses an internal scroll counter for left-boundary clamping (rejects dx
    if counter + dx < 0).  This counter is SEPARATE from SCROLLINFO.nPos.
    ShowScrollBar(FALSE) zeroes BOTH.  SetScrollInfo updates nPos but NOT
    the internal counter.  SOLUTION: use the inHDeliver guard (set TRUE by
    MsbH_DeliverScroll around LVM_SCROLL; WM_NCPAINT and WM_SIZE skip
    Msb_HideNativeBar(H) while the flag is set) so ShowScrollBar(FALSE) never
    fires during delivery.  Both Msb_HideNativeBar(H) and Msb_HideNativeBar(V)
    now use ShowScrollBar(FALSE).  SetWindowLongPtr(~WS_HSCROLL) was tried
    (v2026.05.02.10) but only clears the style bit \u2014 the NC area slot stays
    visible as a 3-px strip when the custom H-bar is shown.  inHDeliver makes
    ShowScrollBar(FALSE) safe for H-axis too.  Also pre-seed
    SetScrollInfo(nPos=lvHPos, FALSE) immediately before LVM_SCROLL so any
    internal ShowScrollBar(TRUE) within LVM_SCROLL re-seeds from the correct
    baseline.  Both left and right tilt-wheel work reliably.
  - sqlite3 amalgamation version: pin to the version used in SetupCraft
    at publish time and note it here.
  - IsDialogMessageW gotcha: ONLY call IsDialogMessageW on windows that were
    actually created via DialogBox/CreateDialog.  On WS_POPUP windows built
    manually, IsDialogMessageW consumes the first WM_LBUTTONDOWN on any
    non-focused child to set keyboard focus (returns TRUE without forwarding
    the click) — button only fires on second click.  Handle Escape via an
    explicit WM_KEYDOWN VK_ESCAPE case in the wndproc instead.
  - RichEdit clickable URL pattern: send EM_AUTOURLDETECT TRUE to auto-style
    URLs with CFE_LINK.  In EN_LINK handle WM_LBUTTONDOWN — extract text via
    EM_GETTEXTRANGE on pLink->chrg, open with ShellExecuteW(hwnd, L"open", ...).
    Hand cursor on hover: subclass the RichEdit, intercept WM_SETCURSOR, call
    EM_CHARFROMPOS to get the char index under the cursor, then EM_GETCHARFORMAT
    (SCF_SELECTION after EM_EXSETSEL to that char) to read CHARFORMAT2W.
    If dwEffects & CFE_LINK, SetCursor(IDC_HAND) and return TRUE — before
    DefWindowProc can override it.  Restore selection after the check.
    with CreateWindowExW it consumes the first WM_LBUTTONDOWN on a non-focused
    child control (to set keyboard focus) and returns TRUE without generating
    BN_CLICKED — the button appears broken until it already has focus.  It also
    triggers spurious WM_SETFOCUS/WM_KILLFOCUS messages that cause plain
    BS_PUSHBUTTON hover states to blink.  For manual message loops on popup
    windows, handle Tab/Return/Escape explicitly in WM_KEYDOWN instead.
  - Tooltip hide-before-modal gotcha: always call HideTooltip() BEFORE calling
    EnableWindow(parent, FALSE) or entering a modal message loop.  The modal
    loop disables the parent window so WM_MOUSELEAVE is never posted to the
    hovered control — the tooltip stays visible for the entire modal session.
    Pattern: WM_LBUTTONDOWN { HideTooltip(); ShowMyDialog(parent, ...); }
  - Color emoji / Direct2D: GDI DrawTextW with Segoe UI Emoji renders only
    the monochrome outline path — GDI does not support OpenType COLR color
    tables.  For full-color emoji use ID2D1HwndRenderTarget::DrawText with
    D2D1_DRAW_TEXT_OPTIONS_ENABLE_COLOR_FONT and an IDWriteTextFormat created
    against "Segoe UI Emoji".  Link d2d1 + dwrite.  Convert physical pixels to
    DIPs for D2D coords: Dip(physPx) = physPx * 96.0f / dpi.  Follow the
    CreateDeviceResources / DiscardDeviceResources pattern and handle
    D2DERR_RECREATE_TARGET in EndDraw.  On WM_SIZE call RT->Resize(SizeU(w,h)).

  - UpDown spinner (UDM_SETPOS32) dirty-flag trap: UDM_SETPOS32 triggers
    EN_CHANGE on the buddy edit.  If programmatic updates happen during window
    creation (e.g. populating spinners from saved values), the EN_CHANGE handler
    fires with a valid value and calls MarkAsModified() before the user has
    touched anything.  Guard pattern: set a bool ignoring = true around every
    programmatic UDM_SETPOS32 / SetWindowText call and check it at the top of
    the EN_CHANGE handler with an early return.

  - Modal message loop + TreeView repaint trap: the modal loop for a WS_POPUP
    dialog must guard with IsWindow in the while condition:
        while (IsWindow(hDlg) && GetMessageW(&msg, NULL, 0, 0) > 0) { ... }
    NEVER follow the loop with a PeekMessage(PM_REMOVE) drain — it consumes
    WM_PAINT messages queued by TreeView_Expand / TreeView_SelectItem, leaving
    the tree blank until the user clicks on it.  To force a synchronous repaint
    after all tree modifications, call:
        InvalidateRect(hTreeView, NULL, TRUE);
        UpdateWindow(hTreeView);

  - Measure-window dirty-flag trap: If a temporary measurement window (e.g. a
    off-screen RichEdit for height measurement) triggers EN_CHANGE on ANY shared
    handler, MarkAsModified() fires unintentionally.  Always create measurement
    windows without an ID (NULL HMENU) or give them a unique ID range that the
    EN_CHANGE dispatcher ignores.

  - Cancel-guard snapshot timing: When a preview/modal dialog takes a snapshot of
    "initial state" to detect user changes (for a "discard changes?" prompt on
    cancel), the snapshot must be captured AFTER all programmatic layout/auto-fit
    operations complete — not before them.  Taking it before means every open
    looks "changed" and the discard prompt fires spuriously.

  - Duplicate-name rename sub-dialog pattern (Scripts / any list-with-names page):
    Instead of blocking the user with a plain validation error on a name conflict,
    open a compact sub-dialog (ShowRenameDialog / RenameDlgProc) with a conflict
    message, a pre-filled all-selected name edit, and Rename/Cancel buttons.
    Validate the new name for empty AND still-taken inside RenameDlgProc; both
    errors use ShowValidationDialog.  On success write the new name back into the
    working row struct and let the outer OK handler proceed.  See scripts_INTERNALS.txt
    §Duplicate-name detection and rename sub-dialog for the reference implementation.

  - Scintilla runtime-load pattern (no import lib required):
    1. LoadLibraryW(L"Scintilla.dll") + LoadLibraryW(L"Lexilla.dll")
    2. GetProcAddress(hSciDll, "Scintilla_RegisterClasses")(hInst) — registers the
       "Scintilla" WndClass.
    3. GetProcAddress(hLexDll, "CreateLexer") — store as (void*(*)(const char*)).
    4. CreateWindowExW(WS_EX_CLIENTEDGE, L"Scintilla", ...) to create the editor.
    5. SendMessage(hSci, SCI_SETILEXER, 0, (LPARAM)pfnCreateLexer("batch"|"powershell"))
       to attach the lexer.
    If either DLL is unavailable (SciLoad returns false), fall back to a plain
    multiline WS_BORDER EDIT control so the feature degrades gracefully.

  - Win32 GUID generation (CoCreateGuid + StringFromGUID2):
    #include <objbase.h>  (link uuid, ole32 — already in most Win32 projects)
    GUID guid; CoCreateGuid(&guid);
    WCHAR buf[40]; StringFromGUID2(guid, buf, 40);
    // buf = L"{XXXXXXXX-XXXX-XXXX-XXXX-XXXXXXXXXXXX}"  (38 chars + NUL)
    // Trim braces if needed: std::wstring s(buf+1, buf+37);
    Good for: Inno Setup AppId GUIDs, install tokens, unique keys.
    CoCreateGuid uses the OS CSPRNG — cryptographically adequate for identifiers.

  - TTM_ADDTOOL is unreliable for owner-drawn / icon-only buttons:
    Standard Win32 tooltips (TTM_ADDTOOL with TTF_IDISHWND | TTF_SUBCLASS) work
    well for text-label buttons but break silently for owner-drawn buttons with no
    window text (isIconOnly path).  The subclass approach is robust:
    1. Subclass the button HWND with SetWindowLongPtrW(GWLP_WNDPROC, myProc).
    2. In WM_MOUSEMOVE: if (!IsTooltipVisible()) ShowMultilingualTooltip(...);
       if (!trackingFlag) { TrackMouseEvent(TME_LEAVE); trackingFlag=true; }
    3. In WM_MOUSELEAVE: HideTooltip(); trackingFlag=false;
    4. In WM_NCDESTROY: restore the original proc.
    This is identical to the pattern used for shell icon tooltips (WarningIcon,
    CompInfoIcon, RegenBtn) throughout SetupCraft.

  - Shared subclass proc for many dialog controls via window property (avTip pattern):
    When a dialog has many checkboxes/radios that each need their own tooltip text,
    use SetPropW to attach the tip string to each HWND and a single shared subclass
    proc to read it back.
    1. Before subclassing:
         SetPropW(hCtrl, L"avTip", (HANDLE)(const wchar_t*)L"Your tooltip here");
    2. Subclass all controls with the same proc:
         s_avPrevChkProc = (WNDPROC)SetWindowLongPtrW(hCtrl, GWLP_WNDPROC, (LONG_PTR)AVFlag_SubclassProc);
    3. In the shared proc WM_MOUSEMOVE:
         const wchar_t* tip = (const wchar_t*)GetPropW(hwnd, L"avTip");
         if (tip) ShowMultilingualTooltip({ {L"", tip} }, rc.left, rc.bottom+4, GetParent(hwnd));
    The s_avPrevChkProc static is the same for all controls because they share the
    same underlying BUTTON class window proc.  Win32 TOOLTIPS_CLASS (TTM_ADDTOOL)
    does not work in dynamically-created dialogs — it never fires TTN_GETDISPINFO
    for checkboxes without WS_EX_NOPARENTNOTIFY set and never over owner-drawn
    controls.  The subclass+property pattern is reliable and requires no extra
    cooperation from the parent dialog proc.

  - Live syntax hint label below type-dependent input fields:
    When a dialog has a "type" combo and a "data" edit whose expected format varies
    by type (e.g. registry value types: DWORD wants decimal/hex, BINARY wants hex
    bytes, SZ wants plain text), add a narrow gray STATIC label immediately below the
    edit and update it on CBN_SELCHANGE.
    Pattern (AddValueDialogProc):
    1. Create the STATIC at creation time with the hint for the initial type:
         CreateWindowExW(0, L"STATIC", GetRegTypeHint(initType),
             WS_CHILD | WS_VISIBLE | SS_LEFT | SS_NOPREFIX,
             editX, dataY + S(ROW_H) + S(GAP_DH), editW, S(HINT_H),
             hwnd, (HMENU)IDC_HINT, hInst, NULL);
    2. In WM_CTLCOLORSTATIC, return gray for the hint control:
         if (GetDlgCtrlID((HWND)lParam) == IDC_HINT) SetTextColor(hdc, RGB(120,120,120));
    3. In WM_COMMAND / CBN_SELCHANGE, update the label:
         SetWindowTextW(GetDlgItem(hwnd, IDC_HINT), GetRegTypeHint(selectedType));
    4. Include the hint row in the clientH calculation so the window is sized right:
         clientH += S(GAP_DH) + S(HINT_H);
    The hint text function is a simple switch/if-chain returning static string literals
    — no allocation needed and safe to pass directly to SetWindowTextW.

  - Multi-select TreeView inline picker pattern (read-only EDIT + "…" button + modal tree):
    When a field must contain a space-separated set of names drawn from a hierarchical
    data source, replace the plain EDIT with:
    1. An ES_READONLY edit narrowed by (pickBtnW + gap).
    2. A small "…" BUTTON immediately to the right (IDC_PICK, AV_PICK_W = 30).
    3. A modal PickerDialogProc containing a TVS_FULLROWSELECT | TVS_HASBUTTONS |
       TVS_SHOWSELALWAYS TREEVIEW with the system image list for real file icons.
    Construction:
    - Attach system small-icon list: SHGetFileInfoW(..., SHGFI_SYSICONINDEX|SHGFI_SMALLICON).
      TreeView_SetImageList(hTree, hSys, TVSIL_NORMAL). Shell owns the list; never destroy it.
    - Per-item icon: sysIconIdx(path, attr) — try real path first; SHGFI_USEFILEATTRIBUTES fallback.
    - TVIF_PARAM | TVIF_IMAGE | TVIF_SELECTEDIMAGE on insert; section headers lParam=-1.
    - Pre-highlight items by matching against the current space-split token set.
    - Expand all section and folder nodes after population.
    Multi-select via PickMultiCtx + subclass:
      struct PickMultiCtx { std::set<HTREEITEM> selected; HTREEITEM anchor; WNDPROC origProc; };
      // stored as TreeView property L"hPickCtx"; deleted in WM_NCDESTROY.
      Subclass proc intercepts WM_LBUTTONDOWN:
        - plain click   → selected = { hit }; anchor = hit
        - Ctrl+click    → toggle hit; anchor = hit; return 0 (suppress default)
        - Shift+click   → walk TreeView_GetNextVisible to build range lo..hi from anchor;
                          selected = all non-header items in range; return 0
        - header click  → clear selected if plain click
      WM_KEYDOWN 'A' + Ctrl → select all non-header visible items.
      WM_NCDESTROY → RemoveProp, delete ctx, restore origProc, call origProc.
    Highlight via NM_CUSTOMDRAW in parent:
      case CDDS_PREPAINT: return CDRF_NOTIFYITEMDRAW;
      case CDDS_ITEMPREPAINT:
        if (ctx->selected.count(cd->nmcd.dwItemSpec)) {
          cd->clrTextBk = GetSysColor(COLOR_HIGHLIGHT);
          cd->clrText   = GetSysColor(COLOR_HIGHLIGHTTEXT);
          return CDRF_NEWFONT; }
    Collection on OK:
      void CollectSelected(HWND hTree, HTREEITEM h, const set<HTREEITEM>& sel, wstring& out) {
          while (h) {
              if (sel.count(h)) { TVITEMW tv={TVIF_PARAM,h}; TreeView_GetItem(hTree,&tv);
                  if (tv.lParam>=0) out += source[tv.lParam].name + L' '; }
              CollectSelected(hTree, TreeView_GetChild(hTree,h), sel, out);
              h = TreeView_GetNextSibling(hTree,h); } }
    Buttons: CreateCustomButtonWithIcon (Green OK imageres#89 / Red Cancel shell32#131)
             + WM_DRAWITEM handler that calls DrawCustomButton.
    Window: WS_POPUP | WS_CAPTION | WS_SYSMENU, WS_EX_DLGMODALFRAME.
    Modal loop: EnableWindow(parent,FALSE) → GetMessageW pump → EnableWindow(parent,TRUE).
    Size: fixed (e.g. 400×500) — tree has a scrollbar.
    Positioning: try right of "…" button; clamp to work area; fall back to centred.
    NOTE: The earlier version used TVS_CHECKBOXES with INDEXTOSTATEIMAGEMASK for the
    state images. That approach works but does not allow multi-select; it is superseded
    by the subclass pattern above for all new pickers.
