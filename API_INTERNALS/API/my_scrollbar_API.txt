1. PURPOSE
──────────
  Drop-in custom scrollbar for any SCROLLINFO-based Win32 window, with full
  support for RichEdit controls.  Replace the native bar by calling
  msb_attach(); everything else — sizing, painting, interaction, DPI scaling —
  is handled internally.

  Features
  ─────────
  • Vertical and/or horizontal bar, independently attached.
  • Proportional thumb derived from SCROLLINFO (or measured content for
    RichEdit, because EM_SHOWSCROLLBAR stops the control maintaining nMax/nPos).
  Fade / hidden mode: bar collapses to a 3 px strip when idle, expands on
    cursor proximity.  MSB_NOHIDE keeps the bar fully visible at all times.
  • Click-to-position on the track; auto-repeat on arrow and track hold.
  • Full thumb drag (SB_THUMBTRACK / SB_THUMBPOSITION / SB_ENDSCROLL).
  • Own WM_MOUSEWHEEL handler (respects SPI_GETWHEELSCROLLLINES; sub-tick
    accumulation for precision devices).
  • EM_SETRECT format-rect management for RichEdit: shrinks the usable height
    so the last line is always scrollable fully above the H bar, with a
    breathing-room margin (MSB_VERT_MARGIN, 6 logical px) below it.
  • DPI-aware: all pixel values go through S() internally; works at any DPI.
  • No SetupCraft dependencies — only <windows.h>.

                          !! GOLDEN RULES FOR LISTVIEW H-AXIS (hard-won) !!

                          1. LVM_SCROLL fires WM_NCPAINT + WM_HSCROLL
                             SYNCHRONOUSLY BEFORE it returns.  WM_NCPAINT fires
                             BEFORE the header control repositions its items.
                             Therefore Msb_GetListViewHPos() (reads header rect)
                             returns the OLD position inside the scroll hot-path.
                             NEVER call it for delta calculations inside any path
                             that is reachable from LVM_SCROLL.

                          2. After LVM_SCROLL: set ctx->lvHPos = newPos (clamped
                             target).  Do NOT re-read the header — it has not
                             moved yet.  lvHPos self-corrects at drag-start
                             (WM_LBUTTONDOWN before SetCapture) and in
                             msb_reposition / WM_SIZE where the ListView has
                             fully settled.

                          3. Delta calculation: int delta = newPos - ctx->lvHPos.
                             Both sides are our own tracked values; the result is
                             always correct regardless of header state.

                          4. SAFE places to call Msb_GetListViewHPos():
                             drag-start (before SetCapture), msb_reposition,
                             WM_SIZE (with capture guard).  Nowhere else.

                          5. msb_attach BEFORE population (RefreshList /
                             ListView_InsertItem) so the WM_NCPAINT subclass is
                             in place during row insertion — prevents blink.

                          6. After RefreshList: ShowScrollBar(FALSE) + msb_sync
                             ONLY.  No capture/restore needed — msb_sync calls
                             Msb_GetListViewHPos while the ListView is idle.

                          7. Line step = S(ctx, 20) logical px.  Using
                             ListView_GetColumnWidth(col0) is wrong — at 35 %
                             list width that is ~280 px and feels like track-click
                             rather than arrow.

                          8. Msb_Scroll ListView H MUST call Msb_Layout +
                             InvalidateRect + UpdateWindow before returning, even
                             via the early-return path — otherwise the thumb never
                             moves visually even though the content scrolled.
    TreeView V+H       ✓  Show/hide, scrollwheel, thumb drag all working.

  #include "my_scrollbar.h"

3. PUBLIC API
─────────────

  typedef void* HMSB;

  ── msb_attach ──────────────────────────────────────────────────────────────
  HMSB msb_attach(HWND hTarget, DWORD flags);

    Attach a custom scrollbar to hTarget.

    flags   One or more of:
              MSB_VERTICAL    (0x00)  Bar on the right edge  [default]
              MSB_HORIZONTAL  (0x01)  Bar on the bottom edge
              MSB_NOHIDE      (0x02)  Always fully visible; no fade animation

    Returns an opaque HMSB handle, or NULL on failure.

    Call once per axis.  Attaching both axes to the same target is safe:
      HMSB hV = msb_attach(hWnd, MSB_VERTICAL   | MSB_NOHIDE);
      HMSB hH = msb_attach(hWnd, MSB_HORIZONTAL | MSB_NOHIDE);

    The call automatically:
      • Hides the native scrollbar (EM_SHOWSCROLLBAR / ShowScrollBar) while
        keeping WS_VSCROLL / WS_HSCROLL so SCROLLINFO is maintained.
      • Creates the custom bar as a sibling of hTarget (child of target's
        parent) so it is never scrolled with the target's content.
      • Subclasses hTarget to intercept WM_SIZE, WM_VSCROLL, WM_HSCROLL,
        WM_MOUSEWHEEL, WM_MOUSEHWHEEL, WM_MOUSEMOVE, and WM_DESTROY.
      • For RichEdit targets: applies EM_SETRECT to reserve space below the
        last line (MSB_VERT_MARGIN) and above the H bar.

  ── msb_detach ──────────────────────────────────────────────────────────────
  void msb_detach(HMSB h);

    Detach and destroy the custom scrollbar.
    Restores the native scrollbar and the original window procedure.
    Safe to call with NULL.
    Call from the parent's WM_DESTROY (before child windows are destroyed):

      msb_detach(st->hSbV);  st->hSbV = NULL;
      msb_detach(st->hSbH);  st->hSbH = NULL;

  ── msb_sync ────────────────────────────────────────────────────────────────
  void msb_sync(HMSB h);

    Manually sync the thumb to the current scroll position and repaint.
    Call after any programmatic scroll that bypasses WM_VSCROLL / WM_HSCROLL,
    e.g. after loading new content into a RichEdit that resets the position:

      SendMessageW(hEdit, EM_SETSCROLLPOS, 0, (LPARAM)&pt);
      msb_sync(st->hSbV);

  ── msb_notify_content_changed ──────────────────────────────────────────────
  void msb_notify_content_changed(HMSB h);

    Invalidate the cached content-dimension measurements (richHorzMax,
    richVertMax) and re-sync both the called bar and its peer.
    Call from EN_CHANGE (or equivalent) whenever text is edited so both bars
    reflect the new document dimensions immediately:

      case EN_CHANGE:
          if (st->hSbH) msb_notify_content_changed(st->hSbH);
          break;

    Calling on either bar (V or H) is equivalent — both caches are cleared
    and both bars are synced.

  ── msb_reposition ──────────────────────────────────────────────────────────
  void msb_reposition(HMSB h);

    Re-derive the bar position from the target's current client-rect corners
    and update visibility.  Designed for explicit window-resize events where
    the target's size has changed and the bar must track the new edges.

    Differs from msb_sync in two ways:
      1. Calls Msb_PositionBar (moves the bar window to the new edge location).
      2. Uses the unguarded visibility check (Msb_UpdateVisibility) so the bar
         is hidden if content genuinely fits after the resize — unlike
         msb_sync which also uses Msb_UpdateVisibility — but msb_reposition
         is the correct entry point from WM_SIZE because it also repositions.

    Call from WM_SIZE after EndDeferWindowPos for all four bars:

      msb_reposition(s_hMsbFilesTreeV);
      msb_reposition(s_hMsbFilesTreeH);
      msb_reposition(s_hMsbFilesListV);
      msb_reposition(s_hMsbFilesListH);

    Safe to call with NULL.

  ── msb_set_insets ──────────────────────────────────────────────────────────
  void msb_set_insets(HMSB h, int insetNear, int insetFar);

    Restrict the bar window to a sub-range of the target's edge.
    insetNear : px to skip from the near edge (top for vert, left for horiz).
    insetFar  : px to skip from the far  edge (bottom for vert, right for horiz).
    Use when the target is a top-level window and the bar must not overlap
    fixed chrome (toolbar, status bar):

      msb_set_insets(s_hMsbIdlg, pageY, statusH);

    Triggers an immediate reposition and repaint.

  ── msb_get_bar_hwnd ────────────────────────────────────────────────────────
  HWND msb_get_bar_hwnd(HMSB h);

    Return the bar's HWND.  Use this to exclude the bar from child-moving
    loops — when the target is a top-level window whose scroll handler moves
    all children by -dy, the bar must be skipped or it will drift:

      HWND hMsbBar = msb_get_bar_hwnd(s_hMsbIdlg);
      // ... inside EnumChildWindows or manual loop:
      if (hChild != hMsbBar) { SetWindowPos(hChild, ...); }

  ── msb_set_edge_gap ────────────────────────────────────────────────────────
  void msb_set_edge_gap(HMSB h, int gap);

    Shift the bar inward from the window edge by gap unscaled px.
    Moves a vertical bar left / horizontal bar upward so the hint strip is
    separated from the frame and more visible against the page background:

      msb_set_edge_gap(s_hMsbIdlg, 4);  // 4 px inward on Dialogs page

    Triggers an immediate reposition and repaint.


4. TYPICAL WIRING
─────────────────

  4a. RichEdit with V + H bars (hidden/fade mode)
  ────────────────────────────────────────────────
  // After creating the RichEdit:
  st->hSbV = msb_attach(hEdit, MSB_VERTICAL);
  st->hSbH = msb_attach(hEdit, MSB_HORIZONTAL);

  // In EN_CHANGE (text edited — content dimensions may have changed):
  if (st->hSbH) msb_notify_content_changed(st->hSbH);

  // In WM_DESTROY (before child windows are destroyed):
  msb_detach(st->hSbV);  st->hSbV = NULL;
  msb_detach(st->hSbH);  st->hSbH = NULL;

  4b. RichEdit with V + H bars (always visible — MSB_NOHIDE)
  ────────────────────────────────────────────────────────────
  st->hSbV = msb_attach(hEdit, MSB_VERTICAL   | MSB_NOHIDE);
  st->hSbH = msb_attach(hEdit, MSB_HORIZONTAL | MSB_NOHIDE);

  4c. Top-level page window (vertical bar, restricted to page area)
  ─────────────────────────────────────────────────────────────────
  // On first build of the page:
  if (!s_hMsb)
      s_hMsb = msb_attach(hwnd, MSB_VERTICAL);
  else
      msb_sync(s_hMsb);
  msb_set_insets(s_hMsb, pageY, statusH);  // skip toolbar + status bar
  msb_set_edge_gap(s_hMsb, 4);             // inset 4 px for visibility

  // In SwitchPage / WM_DESTROY — before tearing down children:
  if (s_hMsb) { msb_detach(s_hMsb); s_hMsb = NULL; }

  // In WM_MOUSEWHEEL / WM_VSCROLL child-moving loop — skip the bar:
  HWND hMsbBar = msb_get_bar_hwnd(s_hMsb);
  // EnumChildWindows / manual loop:
  if (hChild != hMsbBar) { SetWindowPos(hChild, ...); }

  // SB_THUMBTRACK — read position from wParam, not GetScrollInfo:
  case SB_THUMBTRACK:
  case SB_THUMBPOSITION: newPos = (int)(short)HIWORD(wParam); break;

  // oldPos must come from your own stored offset, NOT GetScrollInfo:
  //   int oldPos = SC_GetScrollOffset();   // NOT si.nPos
  // Reason: MSB calls SetScrollInfo(nPos=newPos) BEFORE sending
  // SB_THUMBTRACK/SB_THUMBPOSITION, so si.nPos already equals newPos
  // at the time GetScrollInfo runs -> oldPos==newPos -> dy=0 -> no movement.
  // Your stored offset (SC_GetScrollOffset / IDLG_GetScrollOffset) still holds
  // the old value at this point, giving the correct delta.

  Nothing else is needed.  WM_SIZE, scroll events, and mouse wheel are all
  handled transparently inside the subclass.

  !! COMMON MISTAKE — SIF_TRACKPOS and si.nPos do NOT work with MSB !!
  The native scrollbar stores the drag/track position in nTrackPos (read via
  GetScrollInfo SIF_TRACKPOS) and does NOT update nPos until you call
  SetScrollInfo.  MSB is the opposite: it pre-updates nPos via SetScrollInfo
  before sending WM_VSCROLL, and encodes the position in HIWORD(wParam).
  Two consequences:
    1. GetScrollInfo(SIF_TRACKPOS).nTrackPos is stale — use HIWORD(wParam).
    2. GetScrollInfo.nPos == newPos already — use your own stored offset for
       oldPos to compute dy correctly (see wiring pattern above).

  ── Creation style ──────────────────────────────────────────────────────
  WS_VSCROLL | WS_HSCROLL MUST be in the CreateWindowExW style even though
  the native bars will be visually hidden by MSB.  Without these styles the
  TreeView never populates SCROLLINFO (nMax always stays 0), so GetScrollInfo
  returns zero and MSB always reads "no overflow" → bars never appear.
  The visual native bars are suppressed by msb_attach (and by the WM_NCPAINT
  interception — see Implementation notes); the STYLE must remain.

    s_hTree = CreateWindowExW(WS_EX_CLIENTEDGE, WC_TREEVIEWW, L"",
        WS_CHILD | WS_VISIBLE | WS_VSCROLL | WS_HSCROLL |
        TVS_HASLINES | TVS_LINESATROOT | TVS_HASBUTTONS | ...,
        x, y, w, h, hwnd, (HMENU)IDC_MY_TREE, hInst, NULL);

  ── Attach sequence ─────────────────────────────────────────────────────
  Must happen in this exact order:

    // 1. Populate the tree (all InsertItem + Expand calls).
    RebuildMyTree();

    // 2. Install subclass and attach bars.  Initial visibility is evaluated
    //    inside msb_attach via Msb_ContentOverflows.  For the V-axis this
    //    uses a TVGN_NEXTVISIBLE tree-walk (not SCROLLINFO — which is
    //    unreliable at attach time) so bars start in the correct state.
    //    No further WM_SIZE or msb_reposition call is needed after attach.
    s_hMsbTreeV = msb_attach(hTree, MSB_VERTICAL);
    s_hMsbTreeH = msb_attach(hTree, MSB_HORIZONTAL);
    // Raise H-bar above the WS_EX_CLIENTEDGE border so hint strip is visible:
    if (s_hMsbTreeH)
        msb_set_edge_gap(s_hMsbTreeH, GetSystemMetrics(SM_CYEDGE)
                                    + GetSystemMetrics(SM_CYBORDER) + 6);

  ── Teardown (SwitchPage) ───────────────────────────────────────────────
  MUST come BEFORE any loop that calls DestroyWindow on the TreeView.
  If the TreeView's control ID is in controlIds[] or any child-destroy loop,
  DestroyWindow fires WM_DESTROY → internal msb_detach (frees ctx).  An
  explicit msb_detach afterward reuses freed memory → heap corruption → crash.

    // ← Put this ABOVE the controlIds[] DestroyWindow loop.
    if (s_hMsbTreeV) { msb_detach(s_hMsbTreeV); s_hMsbTreeV = NULL; }
    if (s_hMsbTreeH) { msb_detach(s_hMsbTreeH); s_hMsbTreeH = NULL; }
    // ← controlIds[] loop + DestroyWindow calls follow here.

  ── TVN_ITEMEXPANDED hook ───────────────────────────────────────────────
  The TreeView does NOT send WM_VSCROLL or WM_SIZE when items are expanded
  or collapsed.  Hook TVN_ITEMEXPANDED in WndProc WM_NOTIFY.

  !! GUARD IS MANDATORY !!
  RebuildMyTree() calls TreeView_Expand() synchronously, which fires
  TVN_ITEMEXPANDED BEFORE msb_attach has been called.  The guard
  `if (s_hMsbTreeV || s_hMsbTreeH)` is FALSE during initial build (both
  nil) and TRUE after attach — safely skips the handler during build.

  !! JUST CALL msb_reposition — nothing else !!
  Two approaches that DO NOT work and must NOT be used:
  • SendMessageW(WM_SIZE) with unchanged client dimensions: the TreeView
    caches its client size and skips scroll recalculation — V SCROLLINFO
    remains stale, V bar never appears.
  • Capture SCROLLINFO / ShowScrollBar(FALSE) / restore / msb_reposition:
    silently fails for V-axis — TreeView only updates SB_VERT nMax/nPage
    during WM_SIZE, not during expand/collapse, so siV.nMax==0 at
    TVN_ITEMEXPANDED time → restore skipped → V bar never appears.
    H-axis works (H extent set at item insertion time) but V does not.

  msb_reposition() internally calls HideNativeBar → tree-walk
  (Msb_ReconstructTreeVScrollInfo, TVGN_NEXTVISIBLE) → Msb_UpdateVisibility,
  bypassing SCROLLINFO entirely for V and giving the correct answer.

    // In WndProc WM_NOTIFY (inside the main switch, not a sub-dialog):
    if (nmhdr->idFrom == IDC_MY_TREE && nmhdr->code == TVN_ITEMEXPANDED
        && (s_hMsbTreeV || s_hMsbTreeH)) {
        if (s_hMsbTreeV) msb_reposition(s_hMsbTreeV);
        if (s_hMsbTreeH) msb_reposition(s_hMsbTreeH);
    }

  ── WM_SIZE hook ────────────────────────────────────────────────────────
  Call msb_reposition after EndDeferWindowPos for all bars.  msb_reposition
  uses the unguarded Msb_UpdateVisibility so bars correctly go INVISIBLE
  when the window is resized large enough to fit all content.

    // In WM_SIZE, after EndDeferWindowPos for page controls:
    if (s_hMsbTreeV) msb_reposition(s_hMsbTreeV);
    if (s_hMsbTreeH) msb_reposition(s_hMsbTreeH);

  ── ListView re-suppression after populate ──────────────────────────────
  ListView re-enables its native bars every time items are inserted.
  After every PopulateListView / batch InsertItem call:

    if (s_hMsbListV) { ShowScrollBar(s_hList, SB_VERT, FALSE); msb_sync(s_hMsbListV); }
    if (s_hMsbListH) { ShowScrollBar(s_hList, SB_HORZ, FALSE); msb_sync(s_hMsbListH); }

  ── Implementation notes (handled internally — no per-page action needed) ──
  • WM_NCPAINT interception (inNcPaint guard): the TreeView re-enables its
    native scrollbars inside its own WM_NCPAINT handler on every NC repaint
    — triggered by expand/collapse, scrollwheel, focus, hover, anything.
    The subclass intercepts WM_NCPAINT: lets origProc paint the NC area,
    then immediately calls HideNativeBar for both axes.  The resulting
    re-entrant WM_NCPAINT (from ShowScrollBar(FALSE)) is forwarded straight
    to origProc via the `inNcPaint` flag and then cleared.  Result: native
    bars are permanently invisible regardless of what triggers NC repaint.
  • V-axis overflow via TVGN_NEXTVISIBLE tree-walk: SCROLLINFO.nPage is
    unreliable for TreeView V-axis — ShowScrollBar(FALSE) zeroes it, and
    WM_SIZE with unchanged client rect may not recompute it.  V-axis
    overflow detection walks expanded visible items via TVGN_NEXTVISIBLE
    from root, counting rows against (clientHeight / itemHeight) capacity.
    Msb_ReconstructTreeVScrollInfo (called by msb_reposition and in every
    HideNativeBar path for V) also rebuilds a synthetic SCROLLINFO
    (nMin=0, nMax=totalRows-1, nPage=capacity, nPos=firstVisibleRowIndex)
    so thumb position, arrow clicks, and thumb drag all work correctly.
  • H-axis uses capture/restore (not tree-walk): H extent is updated
    immediately on item insertion, so GetScrollInfo(SB_HORZ) is reliable.
    Capture before HideNativeBar, restore after, in all five event handlers.
  • WM_SIZE re-entrancy (inWmSize guard): HideNativeBar → ShowScrollBar(FALSE)
    changes the NC area → Windows fires synchronous WM_SIZE back to the target.
    Guard returns 0 WITHOUT calling origProc (origProc would re-show native bars).
  • Scroll events (WM_VSCROLL, WM_HSCROLL, WM_MOUSEWHEEL) call HideNativeBar
    + Msb_ReconstructTreeVScrollInfo after origProc, then the GUARDED
    visibility check (Msb_UpdateVisibilityGuarded) — allows INVISIBLE→HIDDEN
    on first overflow, blocks HIDDEN→INVISIBLE from transient zero-range
    events (e.g. auto-scroll-to-selection).  msb_reposition uses the
    UNGUARDED check — only call from real layout events (WM_SIZE,
    TVN_ITEMEXPANDED).
  • ListView: H-axis scroll position is tracked independently in
    MsbCtx.lvHPos because GetScrollInfo(SB_HORZ).nPos is unreliable —
    ShowScrollBar(FALSE) zeroes it and LVM_SCROLL does not update it.
    Arrow buttons use WM_HSCROLL (which the ListView handles correctly when
    nMax is pre-restored via SetScrollInfo + Msb_FixListViewHScrollInfo);
    WM_HSCROLL interceptor captures nPos after origProc and writes it to
    lvHPos before HideNativeBar zeroes SCROLLINFO.  Track-click and thumb-
    drag call LVM_SCROLL with delta = newPos - lvHPos; update lvHPos = newPos.
    nMax/nPage for thumb sizing come from Msb_FixListViewHScrollInfo: live
    column widths (Msb_ListViewTotalColumnWidth) and GetClientRect.
    nPos for Msb_Layout also uses lvHPos.
  • TreeView thumb drag: SetScrollInfo + SB_THUMBTRACK live; SB_THUMBPOSITION
    on mouse-up only.

  Colors

6. RICHEDIT SPECIFICS
──────────────────────
  RichEdit stops maintaining SCROLLINFO (nPos, nMax) when its native bar is
  hidden via EM_SHOWSCROLLBAR.  my_scrollbar works around this:

  Vertical nMax   Measured via Msb_MeasureRichVertMax: EM_GETLINECOUNT +
                  EM_LINEINDEX + EM_POSFROMCHAR + EM_GETSCROLLPOS.  Cached
                  in MsbCtx.richVertMax; cleared on msb_notify_content_changed.

  Horizontal nMax Measured via Msb_MeasureRichHorzMax: scans every line's
                  right edge via EM_POSFROMCHAR(firstChar + lineLen) +
                  EM_GETSCROLLPOS.  Cached in MsbCtx.richHorzMax.

  nPos            Read via EM_GETSCROLLPOS (always accurate even when the
                  native bar is hidden).

  Format rect     EM_SETRECT sets the RichEdit formatting rectangle to
                  (0, 0, w, ch - hBarH - MSB_VERT_MARGIN) so that:
                    • The last line is scrollable fully above the H bar.
                    • MSB_VERT_MARGIN px of white space appear below it.
                  Re-applied after every WM_VSCROLL and WM_HSCROLL because
                  EM_SHOWSCROLLBAR causes RichEdit to reset the format rect.

  Native bar      Suppressed with EM_SHOWSCROLLBAR (not ShowScrollBar —
  suppression     the latter does not stick on RichEdit).  Re-suppressed
                  after every WM_SIZE, WM_VSCROLL, and WM_HSCROLL.
