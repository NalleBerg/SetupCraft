1. PURPOSE
──────────
  Isolated horizontal-scroll delivery for all controls (ListView, TreeView,
  RichEdit).  Rewritten from scratch so H-scroll can be improved without any
  risk of disturbing the working vertical scrollbar.

  Architecture: my_scrollbar_hscroll.cpp is #included directly inside
  my_scrollbar_vscroll.cpp.  It is NOT a separate compilation unit.
  It has full access to MsbCtx, S(), Msb_FixListViewHScrollInfo, and all
  other internal helpers from my_scrollbar_vscroll.cpp.

  All H-scroll calls from Msb_ScrollToPos route through MsbH_DeliverScroll
  (defined in my_scrollbar_hscroll.cpp), regardless of control type.


2. INCLUDE
──────────
  Files that attach an H-bar include BOTH headers:

    #include "my_scrollbar_vscroll.h"   // full API declaration
    #include "my_scrollbar_hscroll.h"   // documents intent; pulls vscroll.h

  Files that only use V-scroll include only my_scrollbar_vscroll.h.

4. GOLDEN RULES FOR LISTVIEW H-AXIS
─────────────────────────────────────
  1. lvHPos is AUTHORITATIVE.  Only written by:
       a. WM_LBUTTONDOWN drag-start (header idle — correct ground truth)
       b. MsbH_DeliverScroll()      (sets lvHPos = clamped newPos)
       c. WM_SIZE / msb_reposition  (layout settle — header correct)
     NEVER re-read from the header mid-drag / mid-scroll.
     Reason: LVM_SCROLL fires WM_NCPAINT BEFORE the header repositions.
     Msb_GetListViewHPos() inside the scroll hot-path returns the OLD
     position → delta always computed from 0 → scroll never accumulates.

  2. LVM_SCROLL(dx, 0) is the only reliable H-scroll primitive.
     Before calling it, set SCROLLINFO.nPos = lvHPos via SetScrollInfo
     (FALSE = no-redraw; does not post WM_NCPAINT) so the left-boundary
     clamp (nPos + dx >= 0) has the correct baseline.  Also set
     inHDeliver = TRUE around the call (see Rule 4).
     HOWEVER: LVM_SCROLL's actual clamp uses an internal counter, not
     SCROLLINFO.nPos directly.  SetScrollInfo alone cannot restore that
     counter.  See Rule 4 for the full picture.

  3. WS_HSCROLL is removed by ShowScrollBar(FALSE) in the WM_NCPAINT
     intercept.  LVM_SCROLL works regardless of WS_HSCROLL — it does NOT
     check the style bit before scrolling.

  4. LVM_SCROLL is a DELTA scroll.  It uses an internal scroll counter for
     left-boundary clamping (rejects dx if counter + dx < 0).  This counter
     is separate from SCROLLINFO.nPos.  ShowScrollBar(FALSE) zeroes BOTH
     the counter and nPos.  SetScrollInfo updates nPos but NOT the counter.
     Result: between deliveries, the counter is always 0 even though lvHPos
     may be > 0 — leftward deltas are rejected.
     Current approach (in code): SetScrollInfo(nPos=lvHPos, FALSE) + inHDeliver
     guard.  This prevents ShowScrollBar(FALSE) during the scroll itself, but
     cannot restore the counter to lvHPos before the call.  Right works; left
     still broken.
     Next fix direction (see here.txt): set inHDeliver = TRUE FIRST, then
     call ShowScrollBar(SB_HORZ, TRUE).  ShowScrollBar(TRUE) should
     re-initialize the ListView's internal counter from SCROLLINFO.nPos.
     Because inHDeliver is already TRUE, the WM_NCPAINT that ShowScrollBar(TRUE)
     triggers will skip Msb_HideNativeBar — so ShowScrollBar(FALSE) never
     fires and the counter stays at lvHPos for LVM_SCROLL.
