  The SINGLE critical bug introduced in this session's working tree:

  In WM_MOUSEMOVE, inside the "if (ctx->dragging)" block, these TWO lines
  were added at the start of drag processing:

      if (ctx->isListView && !vert)
          ctx->lvHPos = Msb_GetListViewHPos(ctx->hTarget);

  These appear just before:
      int cur   = vert ? GET_Y_LPARAM(lParam) : GET_X_LPARAM(lParam);

  This violates GOLDEN RULE 1 (see my_scrollbar_API.txt and
  scrollbar_INTERNALS.txt): WM_MOUSEMOVE fires while we hold capture,
  which means it fires inside the scroll hot-path.  But worse: LVM_SCROLL
  does NOT actually move the header when WS_HSCROLL is absent.  So after
  every Msb_ScrollToPos() call, ctx->lvHPos is correctly set to newPos,
  but on the VERY NEXT WM_MOUSEMOVE the re-read sets ctx->lvHPos = 0
  (header hasn't moved).  On the next iteration:
      delta = newPos - ctx->lvHPos = newPos - 0 = same large value again
  This makes every drag step recompute from scratch against 0, so the
  scroll position never accumulates.

  The debug log confirms this exactly:
      LBUTTONDOWN: vert=0 clickPos=340 lvHPos=0 (correct, read from header)
      ScrollToPos: lvHPos=0 newPos=2 delta=2
      ScrollToPos: post-scroll lvHPos=2 header=0  <- LVM_SCROLL did nothing
      ScrollToPos: lvHPos=0 newPos=2 delta=2       <- lvHPos reset to 0!
      ScrollToPos: post-scroll lvHPos=2 header=0
      (repeats forever -- cursor barely moved, newPos=2 every step)

  The re-read is the only change that could cause this -- it was not in HEAD.

  ── THE FIX (ready to apply in Norway) ─────────────────────────────────────

  In my_scrollbar.cpp, WM_MOUSEMOVE handler, REMOVE these two lines:

      if (ctx->isListView && !vert)
          ctx->lvHPos = Msb_GetListViewHPos(ctx->hTarget);

  After removing them, lvHPos will only be updated by:
    1. WM_LBUTTONDOWN (drag-start): reads from header (correct, ListView idle)
    2. Msb_ScrollToPos: sets lvHPos = newPos (the clamped target)
    3. WM_SIZE / msb_reposition: reads from header (ListView settled)
  This is exactly what HEAD = 8ecadde does.

  ── SECONDARY QUESTION: why does LVM_SCROLL not move the header? ─────────────

  LVM_SCROLL fires WM_NCPAINT synchronously, which calls ShowScrollBar(FALSE),
  which removes WS_HSCROLL from the ListView's window style.  Next time
  LVM_SCROLL is called, it checks WS_HSCROLL and silently does nothing when
  the flag is absent.  The header never moves.  ctx->lvHPos = newPos is the
  only tracking mechanism -- the header is only a ground-truth for DRAG-START
  and RESIZE events, not mid-drag.

  If the fix above still does not scroll the content visually, the next thing
  to investigate is whether WS_HSCROLL is indeed absent when LVM_SCROLL is
  called.  Diagnostic: log GetWindowLongW(ctx->hTarget, GWL_STYLE) & WS_HSCROLL
  inside Msb_ScrollToPos.  If absent, add ShowScrollBar(TRUE) before the call
  with a guard so the WM_SIZE it triggers does not reset lvHPos.
