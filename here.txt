 3. Drag-and-drop — TreeView folder drag and ListView file drag

   CURRENT SYMPTOM (end of day):
     After the second attempt at fixing it, neither folders nor
     files react to drag at all. The cursor never changes and no
     drop-target highlighting appears.

   HISTORY OF ATTEMPTS:
     Attempt 1 (original):
       Used TVN_BEGINDRAG / LVN_BEGINDRAG + ImageList_DragEnter()
       for a ghost image.
       Symptom: Folder got a partial blue flicker/artifact.
       Files did not react.

     Attempt 2 (today's fix):
       Removed all ImageList ghost code (DWM compositing makes
       ImageList_DragEnter incompatible with modern Windows).
       Added WM_PARENTNOTIFY to detect LMB press in child controls,
       then tried to detect drag threshold in the parent's
       WM_MOUSEMOVE.
       Symptom: Nothing reacts at all now.

   ROOT CAUSE ANALYSIS:
     The fundamental problem is WHERE mouse messages are delivered:

     A) Before SetCapture(hwnd) is called:
        - LMB press lands in TreeView/ListView child window.
        - Those child controls internally take mouse capture.
        - Subsequent WM_MOUSEMOVE messages go to the CHILD, not
          the parent (hwnd). So the parent's WM_MOUSEMOVE drag-
          threshold check never runs until the cursor leaves the
          child entirely.
        - TVN_BEGINDRAG / LVN_BEGINDRAG fire from the child's
          internal handling (after the child decides "this looks
          like a drag"), but:
            - TVS_EDITLABELS delays item selection (waits for
              double-click timeout to know if user wants to edit).
              This may delay or suppress TVN_BEGINDRAG.
            - LVN_BEGINDRAG requires the ListView to internally
              detect drag threshold too — if the hit-test isn't
              exactly on the label area, it may not fire.
     
     B) WM_PARENTNOTIFY fallback (attempt 2):
        WM_PARENTNOTIFY(WM_LBUTTONDOWN) fires on the parent, so
        we correctly record the starting point and item. BUT the
        threshold check is in the parent's WM_MOUSEMOVE, which
        doesn't receive messages until hwnd has capture.
        Chicken-and-egg: we need a drag to set capture, and we
        need capture to detect a drag.

   THE CORRECT FIX FOR TOMORROW:
     Subclass both s_hTreeView and s_hListView.
     Give each its own drag-detection subclass proc:

       TreeViewSubclassProc:
         WM_LBUTTONDOWN:
           - Hit-test the click position.
           - If on a non-root folder item: store item + screen pt
             as "potential drag" (s_dragPotTV, s_dragPotPt).
           - Forward to original proc (TreeView needs to select).
         WM_MOUSEMOVE:
           - If LMB still down AND potential drag set:
             - Check SM_CXDRAG/SM_CYDRAG threshold.
             - If exceeded: set s_dragSource, s_dragTreeItem,
               call SetCapture(hwndParent).
             - Parent now receives all WM_MOUSEMOVE until drop.
           - Always forward to original proc.
         WM_LBUTTONUP:
           - Clear potential drag state.
           - Forward to original proc.

       ListViewSubclassProc: same pattern, but check that the
         selected tree node is in s_virtualFolderFiles before
         allowing the drag.

     The main window's WM_MOUSEMOVE, WM_SETCURSOR, WM_LBUTTONUP,
     and WM_CAPTURECHANGED handlers are then triggered correctly
     once SetCapture(hwndParent) is called from the subclass.

   IMPLEMENTATION STEPS:
     1. In mainwindow.h:
        - Declare static LRESULT CALLBACK TreeDragSubclassProc(...)

     2. In mainwindow.cpp:
